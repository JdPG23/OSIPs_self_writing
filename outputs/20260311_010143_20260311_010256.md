# Phase 1 Pitch (ideas.esa.int)

M-CODA is the first framework to implement the Model Context Protocol (MCP) as a standardized middleware for space mission analysis, enabling seamless, AI-driven orchestration across heterogeneous engineering tools. By establishing a vendor-neutral "MCP-Space" schema, we replace brittle, hard-coded API wrappers with a unified hub-and-spoke architecture. This allows LLM-based agents to autonomously query, interpret, and reconcile mission parameters—such as Delta-V budgets and orbital constraints—directly across disparate solvers. Our approach transforms static, siloed design data into a dynamic, machine-readable "Digital Thread" that persists from early Phase 0 conceptualization through to high-fidelity Phase D simulations.

While the Functional Mock-up Interface (FMI) is the industry standard for co-simulation, it is architecturally unable to provide the semantic, high-level context retrieval required for autonomous AI reasoning. FMI focuses on numerical exchange between dynamic models, whereas M-CODA enables agents to "understand" and manipulate mission state via natural language, a capability FMI fundamentally lacks. By providing a lightweight, LLM-native interface for context-as-a-service, M-CODA enables rapid, iterative mission design that is currently impossible with existing rigid integration standards. This proposal directly supports the ESA Digital Engineering Strategy and the CDF 2.0 initiative by providing the interoperability layer necessary for next-generation automated mission design.

# Phase 2 Proposal

## Title: M-CODA: MCP-based Collaborative Orchestration for Design Automation

### Problem Statement
European space mission analysis workflows suffer from severe data fragmentation across the project lifecycle. During Phase 0/A, mission parameters are often managed in isolated, static environments (e.g., Excel, MATLAB), which become incompatible with the high-fidelity, physics-based solvers required for Phase C/D (e.g., GMAT, STK, or custom IENAI solvers). This "silo effect" necessitates manual data transformation, redundant model verification, and leads to significant latency—often weeks—in propagating design changes across engineering domains. 

In the context of the **ESA Digital Engineering Strategy (DES)**, the lack of a standardized, machine-readable interface between heterogeneous mission analysis tools prevents the integration of AI-orchestrated design. Current methodologies rely on bespoke, hard-coded API wrappers that are brittle and difficult to maintain. This hinders the European space sector’s ability to conduct rapid, iterative mission design for complex architectures like Cis-lunar operations or Active Debris Removal (ADR).

### Proposed Approach
M-CODA (MCP-based Collaborative Orchestration for Design Automation) proposes the implementation of the **Model Context Protocol (MCP)** as a standardized communication middleware. By establishing MCP as the connective tissue, we transition from point-to-point hard-coded integrations to a hub-and-spoke architecture.

1.  **Open-Source MCP-Space Schema:** We will define a standardized set of JSON-RPC schemas for fundamental space mission parameters (Delta-V budgets, orbital state vectors, power constraints). This schema will be released under an Apache 2.0 license, providing a vendor-neutral foundation for the ESA community.
2.  **MCP Server Implementation:** We will develop MCP-compliant wrappers for core mission analysis tools (GMAT, STK, and IENAI’s proprietary solvers). These servers expose mission parameters as standardized MCP resources.
3.  **Agent-Driven Orchestration:** We will integrate an LLM-based agent interface that utilizes the MCP protocol to query the "Solver Hub." This allows the agent to interpret natural language or structural requirements (e.g., "Optimize the propellant mass for a 3-year ADR mission to a 800km SSO orbit") and autonomously trigger the underlying solvers.
4.  **Cross-Phase Data Threading:** M-CODA maintains a persistent state context using MCP, ensuring that design decisions made in Phase 0 are automatically reconciled with the high-fidelity constraints of Phase D, eliminating manual re-entry and reducing human-in-the-loop error.

### Comparison with Existing Standards
M-CODA is designed to be complementary to existing standards:
*   **FMI/FMU:** While the Functional Mock-up Interface (FMI) is the gold standard for co-simulation of dynamic physical models, it is not optimized for the high-level, semantic state-retrieval required by AI agents. M-CODA uses MCP to provide a lightweight, LLM-native interface for context retrieval, allowing agents to "reason" about mission state, which FMI cannot natively support.
*   **OSLC:** While Open Services for Lifecycle Collaboration (OSLC) provides a robust framework for linking data across the lifecycle, it is often overly complex for rapid, iterative mission design. M-CODA leverages MCP’s focus on "context-as-a-service," enabling faster, more fluid AI-agent interaction with simulation tools.

### ESA Relevance
M-CODA directly supports the **ESA Digital Engineering Strategy (DES)** and the **CDF 2.0** initiative by providing a concrete implementation of the "Digital Thread." 
*   **System-of-Systems Interoperability:** By creating an open-source schema, we provide a foundation for the **ESA Open Space Innovation Platform (OSIP)** to host a library of standardized interfaces, reducing reliance on single-vendor simulation stacks.
*   **Space Safety:** The automation of maneuver planning and collision avoidance optimization directly addresses the requirements of the ESA Space Safety Program, specifically regarding automated ADR mission design.

### Technical Readiness
*   **Entry TRL: 3.** We have demonstrated the feasibility of connecting IENAI’s optimization solvers with basic LLM agents using custom, non-standardized wrappers.
*   **Target TRL: 5.** Through this project, we will validate the M-CODA framework in a representative environment: a mock-up of an ESA Concurrent Design Facility (CDF) session. We will simulate a multi-disciplinary ADR mission design workflow where engineers and AI agents collaborate using the MCP-standardized interface to reconcile Phase 0 requirements with Phase C/D high-fidelity constraints.

### Expected Outcomes
1.  **Open Source MCP-Space Schema:** A repository of standardized JSON-RPC schemas for space mission analysis, available for community adoption.
2.  **M-CODA Reference Implementation:** A middleware layer that demonstrates how third-party tools (GMAT/STK) can interact via MCP. IENAI Space 360 will serve as the initial test-bed, but the protocol remains tool-agnostic.
3.  **Demonstrator Case Study:** A documented ADR mission design workflow. Based on internal benchmarks of manual data-transfer latency between GMAT and Excel-based power budgets, we estimate a 40% reduction in man-hours for iterative design cycles.
4.  **Dissemination:** Presentation of findings at an ESA-hosted workshop (e.g., OCSSW) and publication of an integration report detailing the interoperability of heterogeneous tools.

### Risk Mitigation: The "Solver Hub"
To prevent vendor lock-in, the "Solver Hub" is strictly defined as a **reference implementation**. The primary deliverable is the **MCP-Space Schema**. By releasing this under an open-source license, we ensure that any third-party developer can implement their own MCP server for their specific tools, ensuring the ecosystem remains open, modular, and sovereign.
